# 可视化语言

训练场共享一套稳定的视觉语义。新增模型或观察视图时，优先复用这些含义，不要按页面重新发明颜色。

| 语义 | 颜色 | CSS 变量 | 常见用途 |
| --- | --- | --- | --- |
| 正类 / 正向贡献 | 绿色 `#22f0a4` | `--viz-positive` | 正类样本、新增有效贡献、通过方向 |
| 负类 / 当前模型 | 蓝色 `#3bd7ff` | `--viz-negative` | 负类样本、当前预测、基线场 |
| 当前关注 | 黄色 `#ffd447` | `--viz-focus` | 阈值、目标、候选切分、当前叶子或支持点 |
| 错误 / 风险 | 红色 `#ff5f57` | `--viz-error` | 残差、错分、违规、过拟合警告 |
| 历史状态 | 灰色 `#a8b0bd` | `--viz-previous` | 上一轮模型、未激活结构、参考轨迹 |

## 线型约定

- 粗实线：当前模型或最终边界。
- 细实线：局部贡献、连接或辅助量。
- 虚线：候选切分、上一轮状态、间隔或尚未应用的结构。
- 半透明色块：概率、归属区域或模型置信度，不单独承担类别信息。

## 页面解释约定

每个训练场必须包含“形态速读”，回答三个问题：

1. **长什么样**：用户应该在画布中识别出的模型几何形态。
2. **本轮看点**：执行一次训练后最值得观察的变化。
3. **失控信号**：超参数或训练过程出现问题时的视觉征兆。

每个观察方式应只承担一个主要问题：模型在哪里、错误在哪里、本轮学了什么，或损失是否下降。

## 可访问性约定

- Canvas 必须包含可读的文本替代。
- 关卡和观察方式容器必须有 `aria-label`。
- 动态训练结论使用 `role="status"` 或 `aria-live`，避免重复播报同一句话。
- 键盘焦点必须清晰；系统请求减少动态效果时，应关闭非必要动画。
